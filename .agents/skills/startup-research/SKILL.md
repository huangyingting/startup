# Startup Research

This file is the workflow narrative: what to run, in what order, with which flags. Two generated companions carry the static reference material; load both into context **in full at session start** (≈45 KB combined — small enough to keep resident, large enough that round-tripping during chapter authoring is wasteful), **`rules.md` first** (it defines the agent policy, ID system, and dimension vocabulary that `contracts.md` then cross-references):

- [`references/rules.md`](references/rules.md) — binding agent policy, gates, ID system, validator dimensions, and figure renderer contracts.
- [`references/contracts.md`](references/contracts.md) — field shapes for the chapter analysis YAML and `report-meta.yaml` (with allowed enum values inline at each field).

Do **not** edit `references/rules.md` or `references/contracts.md` directly. They are generated; change `references/workflow-config.yaml`, `scripts/validation-catalog.mjs`, `scripts/contracts/*`, or renderer contract sources, then rerun the build scripts.

If any prose in this file disagrees with a runtime/script output, **trust the runtime/script output** — that rule binds you, the runtime agent, every time. (Maintainer note, not for runtime: when fixing such a disagreement, edit the source contract — script behavior, schema under `scripts/contracts/`, or `validation-catalog.mjs` — and let the generated `references/*.md` re-render. Do not patch the prose into this SKILL.md.)

## Execution contract

- Run every command from the repository root. All paths in this skill are repo-root relative unless the runtime script prints an absolute path.
- Treat validation commands as gates. Run them directly and preserve full stdout/stderr and nonzero exit codes. Do not pipe gate output through truncating filters.
- `npm run validate` verifies generated references, workflow config, revision graph, historical report contracts, and the website build. It does **not** re-run the current chapter gates against every historical finalized report; `finalize-report.mjs` is the publishability gate for the report currently being authored.
- Prefer `--format json` or `--format compact` on the `check-*` validators **and** the `build-evidence-ledger` / `build-report` assemblers when machine-readable output helps. The repair-hint surfaces escalate from finest- to coarsest-grained — read in this order before guessing:
  - **`issues[].fix`** — per-issue fix string (one per individual failure entry).
  - **`objectFailures[].fixes`** — per-object aggregate (multiple failures on the same id/object collapsed into one fix list; check-chapter only).
  - **`globalHints[].fix`** — chapter-wide root cause (when a single dimension fails N objects, prefer the global fix over N per-issue ones).
  - **`retryOrder[]`** — dimensions ordered root-cause-first; fix in this order.
  - **`suppressedDimensions[]`** — dimensions whose own failures were hidden because a prerequisite dimension failed; they will re-emit after the prerequisite is fixed.
  - The latter four are conditional keys — they only appear when non-empty (see [`references/contracts.md`](references/contracts.md) → *Validation result envelope*).
- The [`hardRules`](references/rules.md) block in *Agent policy (binding)* binds every step: no git command (commit/push/stash/etc.) at any point, no hand-edited `revision:` / `evidence.yaml` / `full-report.yaml` / `summary-card.yaml`, no scratch files outside `.research-cache/<runId>/`, and no edits to a sibling chapter while authoring the current one. Read the block once at session start; it is short.

## Runtime bootstrap

1. Preflight source discovery **before creating a report folder**. Confirm the current runtime has an approved web/search/source-discovery capability for discovering candidate URLs. `fetch-url` is for reviewing direct URLs, not for broad discovery. If no source-discovery capability is available, stop here: in interactive mode ask the user for a source pack of authoritative URLs; in non-interactive/no-ask automation print `source-discovery-unavailable` to stderr and **exit non-zero (use exit code 2 — "intentional skip, no folder created") so the orchestrator does not record the run as a successful no-op**. Do not call `create-report-run.mjs`. If the user only has staged local HTML/PDF/text files, require metadata for each file (`originalUrl`, publisher, title, publication date if known) and cite the original URL; do not invent URLs from filenames.
2. Get the configured chapter list:
   `node .agents/skills/startup-research/scripts/load-chapter-runtime-context.mjs --list`
   Output is the ordered chapter index. Each entry is the full chapter brief (`key`, `order`, `letter`, `file`, `artifact`, `title`, `mission`, `optionalContext` *(list of sibling chapter `key`s whose rollups should be merged into `cumulativeContext` when this chapter is loaded with `--include-context` in sequential mode — not free-form prose)*, `contentRequirements`, `plannedTables`, `plannedFigures`, `evidenceStrategy`, `qualityBar`, `gate`); see [`contracts.md`](references/contracts.md) → *Chapter runtime context shape* for the authoritative shape. The static frame (agent policy, ID system, dimension vocabulary) lives in [`references/rules.md`](references/rules.md). Per-chapter `previousChapter` / `nextChapter` neighbours, `runtimeContext.run.runDate`, `runtimeContext.runCache`, and `runtimeContext.policy` are emitted later by the per-chapter loader call (chapter generation step 1), not by `--list`.
3. Create a report folder:
   `node .agents/skills/startup-research/scripts/create-report-run.mjs "<companyName>" [--website <companyUrl>]`
   - The script writes the **report folder path to stdout** (everything else — hints, refresh-context notices, env-snippet path — goes to stderr). Capture it into `REPORT_FOLDER` so subsequent commands have a stable handle, and **check `$?` immediately before running any other command**; otherwise a failed create can leave `REPORT_FOLDER` empty and the next steps will operate on garbage paths (`basename "" → .`):
     ```sh
     # Drop --website if you don't have a URL; the example brackets are CLI usage notation,
     # not literal shell syntax — keeping them produces an unknown-flag error.
     REPORT_FOLDER=$(node .agents/skills/startup-research/scripts/create-report-run.mjs "<companyName>" --website "<companyUrl>")
     create_status=$?
     if [ "$create_status" -ne 0 ]; then
       echo "create-report-run failed with exit ${create_status}; stop and branch per the exit-code list below" >&2
       # The next step depends on $REPORT_FOLDER, which is empty here; stop
       # this attempt instead of continuing with garbage paths. In a script,
       # `exit "$create_status"` aborts the script; in an interactive shell
       # the equivalent is `return "$create_status"` from a function or just
       # halt the run yourself. Do NOT use a bare `false` — it only sets $?
       # for the if-block and lets the next command run anyway.
       exit "$create_status"
     fi
     ```
   - **Quote multi-word values** in the shell. The company name (e.g. `"Form Energy"`, `"Commonwealth Fusion Systems"`), `--refresh-reason "<text>"`, and any `--website` URL containing shell metacharacters (`?`, `&`, `=`, `#`, `;`, `*`, `(`, `)`, `[`, `]`, `<`, `>`, `|`, `$`, `` ` ``, whitespace) must each be quoted; otherwise the second word is parsed as an unknown flag and the script exits 1.
   - For refresh: add `--refresh --refresh-reason "<refreshReason>"` on the initial create-run invocation. If the script reports multiple current matching reports, list each candidate's runId / `company.name` / `company.website` from `reports/<runId>/summary-card.yaml`, confirm the intended one (in interactive mode, ask the user; in automation, pick the most recent runId), and rerun create-run with `--refresh --refresh-reason "<text>" --refresh-of <priorRunId>` (all three flags — the script rejects `--refresh-of` without `--refresh`) before any refresh context is written. `<priorRunId>` is the **bare runId** (`YYYYMMDDhhmmss-<slug>`, e.g. `20260509143000-acme`), not a folder path — do NOT pass `$REPORT_FOLDER` or `reports/<runId>`. `--resume` reuses the refresh reason cached by the original create-run.
   - Exit `1`: bad CLI arguments or a non-recoverable failure; read stderr (every script prefixes its messages with `[script-name]`) for the offending field, fix it, then rerun. **Refresh-specific exit-1 cases**: (a) `--refresh requested, but no matching finalized report exists for this company/domain` — the company has never been finalized under this name/domain. **Before dropping `--refresh`**, double-check the `<companyName>` and `--website` arguments against `ls reports/` (or the `summary-card.yaml` of recent runs) — a typo or alias mismatch (e.g. `"OpenAI Inc"` vs `"OpenAI"`, `openai.com` vs `chatgpt.com`) silently produces a duplicate fresh run instead of a refresh. Only after confirming the name/domain are correct, drop `--refresh` and rerun as a fresh run. (b) `--refresh requested, but every matching report is already superseded` — the prior chain is broken (the most recent finalized report is not `current`). Stop and surface a user-facing blocker that lists every matching prior runId and its `revision.status` (read each from `reports/<runId>/report-meta.yaml`) so the maintainer can triage; do NOT hand-edit `revision.status` on a prior run to work around it (hardRules forbids editing `revision` fields, and `link-refresh.mjs` will overwrite manual edits anyway). Repair is a maintainer-only operation outside the runtime workflow. (c) `--refresh is ambiguous: multiple current finalized reports match this company/domain` — follow the multi-match recovery in the bullet above (rerun create-run with `--refresh --refresh-reason "<text>" --refresh-of <priorRunId>`). **Resume + refresh consistency cases (also exit 1)**: (d) `cannot resume refresh run <runId> without --refresh` — the original create-run was a refresh and `.research-cache/<runId>/refresh-context.yaml` exists; rerun with `--resume --refresh` (the cached `--refresh-reason` will be reused). (e) `cannot promote in-progress run <runId> into a refresh` — the original was a fresh run; either resume without `--refresh`, or finalize/discard it and start a new `--refresh` run separately.
   - Exit `2`: a finalized duplicate already exists; stop without creating a new report. Fires only on **fresh** runs — refresh runs intentionally accept the duplicate (`refreshTarget` short-circuits the duplicate-risk check). In automation, treat exit 2 as a successful skip only when no report folder was created; surface a duplicate/skip note to the caller instead of retrying with a suffixed folder.
   - Exit `3`: an in-progress folder for the same company slug already exists; rerun with `--resume` (and, if stderr lists multiple candidates, add `--resume-run <runId>`). Do not create suffixed duplicate folders.
   - Exit `4`: no matching in-progress folder exists for `--resume` — rerun **without** `--resume` to create a fresh one.
4. Use `$REPORT_FOLDER` (the path captured above, e.g. `reports/20260509143000-acme`) as `<reportFolder>` for every later command. The runId is the folder basename (`<runId>=20260509143000-acme`). The per-chapter loader (chapter generation step 1) emits `runtimeContext.run.runDate` whenever `--report-folder` is supplied, including on the first chapter and during parallel drafting; that value is the canonical `runDate` for every chapter doc head and for every source-discovery query date token. Never format a date from the model clock. `runDate` is auto-derived UTC `YYYY-MM-DD` from the runId timestamp prefix (set by `create-report-run.mjs` at folder-creation time). The binding rules for using `runDate` as the clock/freshness anchor live in [`rules.md`](references/rules.md) → *Agent policy (binding)* → `researchRules`.
5. Export `STARTUP_FETCH_LOG_PATH` before any `fetch-url` invocation. The default approach:

   ```sh
   RUN_ID=$(basename "$REPORT_FOLDER")
   source ".research-cache/${RUN_ID}/env.sh"
   ```

   - **Default**: source the snippet `create-report-run.mjs` wrote — it sets `STARTUP_FETCH_LOG_PATH=.research-cache/<runId>/_fetch-log.jsonl`.
   - **CI exception**: if `STARTUP_FETCH_LOG_PATH` is already exported (e.g. a workflow-wide trail at `.research-cache/_fetch-log.jsonl`), do **NOT** source the snippet — keep the existing value. `check-chapter` only needs each cited URL to appear in the trail at least once. (`create-report-run.mjs` detects this case and prints `[create-report-run] note: STARTUP_FETCH_LOG_PATH already exported …` instead of the source-the-snippet hint.)
   - **Subagent / isolated-runtime caveat**: if this agent fans work out to subagents that do not inherit shell environment variables (e.g. the `fetch-url` skill is invoked through a subagent runtime), pass the resolved `STARTUP_FETCH_LOG_PATH` value explicitly in each subagent's prompt or context. Use the absolute path (`"$PWD/.research-cache/${RUN_ID}/_fetch-log.jsonl"` from the repo-root shell, since `$PWD` is the repository root per the *Execution contract*) when any subagent may chdir away from the repository root — the relative form only resolves correctly from the repo root. Avoid `realpath` for portability (it is GNU coreutils, not POSIX, and is missing from vanilla macOS and minimal Alpine images).
   - **What `check-chapter` enforces against the trail**: `fetch-url` appends one JSON line per fetch; `check-chapter` emits an `unverifiedSource` **warning** for each cited URL absent from the trail (promoted to a failure under `--strict`). When the env var is unset and no trail file exists at any candidate path, `check-chapter` instead emits a single `fetchTrailMissing` warning so the disabled-audit case is visible rather than silent.

## Chapter generation

**Default drafting mode: parallel.** Spawn every chapter concurrently — wall-clock approaches N× faster (modulo the convergence-rerun pass and any per-chapter retry overhead), the per-chapter spec already keeps chapters loosely coupled, and step 7 below (convergence rerun) collapses the per-chapter dimensions that depend on sibling state (`netNewSources`, `crossChapterRefLeak`) before finalize. Cross-chapter duplicates (`duplicateAnalysisCrossChapter`) and metric/key-fact drift are caught later inside `finalize-report` by `check-cross-chapter`, not by the convergence rerun. Fall back to **sequential** drafting (each chapter starts only after the prior chapter's YAML has landed on disk) only when (a) the agent runtime is single-threaded or token-rate-limited and cannot spawn concurrent workers, or (b) the report has unusually tight chapter coupling that would otherwise force multiple convergence-rerun cycles. Sequential trades wall-clock for cheaper convergence: each later chapter sees prior siblings' rollups via `--include-context` and can avoid `crossChapterRefLeak` / `netNewSources` failures up front, so step 7 is unnecessary.

For each chapter from the `--list` roster:

1. Load its per-chapter delta:
   `node .agents/skills/startup-research/scripts/load-chapter-runtime-context.mjs --order <n> --report-folder <reportFolder> [--include-context]`
   - The loader always emits JSON to stdout; do not pass `--format`.
   - `--order <n>` is the canonical selector; `--key <chapter-key>` and `--file <chapter.file>` are equivalent and useful in retry loops where the failing chapter's key or file path is what you have on hand. Pick one per invocation (parallel workers may use different selectors across invocations).
   - **`--include-context` rule by drafting mode**:
     - **Parallel (default)**: omit `--include-context` on every chapter — any sibling rollup it would project is stale relative to the in-flight peers, and step 7's convergence rerun handles cross-chapter divergence for you.
     - **Sequential (fallback)**: omit `--include-context` on the first chapter (`--order 1` — the rollup would be empty anyway); pass it on every later chapter so the loader projects `contextChapters` and `cumulativeContext` from the already-written siblings.
2. Plan queries → search → fetch → record (run this **before** authoring step 3 so `localEvidence` is populated when the YAML body is written):
   - **`researchQuestions[]` vs `searchQueries[]` are distinct fields with different validators.** `researchQuestions[]` are typed analytical questions with `id` (`Q<L>###`), `type`, `status`, and `targets[]` pointing at `contentRequirements/` / `plannedTables/` / `plannedFigures/` (validated by `researchQuestionShape` / `researchQuestionAnswerCoverage` / `researchQuestionAdverse` / `contentRequirementCoverage`). `searchQueries[]` are the literal search-engine query strings you ran, recorded as audit (validated by `searchQueriesMissing` / `searchQueryFreshness`). Do not store one in the other — the validators look for different shapes.
   - Before issuing web searches, plan source-discovery queries from `runtimeContext.run.runDate`; derive any year/month tokens from that single clock anchor. The binding rules for volatile facts and date tokens live in [`rules.md`](references/rules.md) → *Agent policy (binding)* → `researchRules` and `volatileFactQueryTokens`. Use the host agent's approved web/search capability (or an approved search API/tool provided by the runtime) to discover candidate sources. If this capability is absent, you should already have stopped during the bootstrap preflight; do not invent sources or backfill `localEvidence` from memory.
   - Run searches to discover candidate sources; record each query you actually issued in `localEvidence.searchQueries[]` as provenance/audit, not as the search execution mechanism.
   - For each retained URL, retrieve it with the [`fetch-url`](../fetch-url/SKILL.md) skill (with `STARTUP_FETCH_LOG_PATH` set, this also writes the fetch trail that `unverifiedSource` audits against).
   - Store reviewed sources, atomic claims, typed research questions, and typed evidence gaps in `localEvidence`.
3. Author the chapter YAML at `reportFolder/<runtimeContext.chapter.file>` using:
   - `runtimeContext.chapter` for mission, content requirements, planned tables/figures, quality bar, and gate.
   - [`references/rules.md`](references/rules.md) for agent policy, gates, the **ID system** (mint every `S/C/T/F/Q` id with this chapter's `runtimeContext.chapter.letter`), validator dimensions, and renderer contracts.
   - [`references/contracts.md`](references/contracts.md) for the chapter YAML field shapes and inline allowed enum values.
4. Run the chapter gate (normal pass):
   `node .agents/skills/startup-research/scripts/check-chapter.mjs <reportFolder> <chapter.file> --format json`
   - **Read `warnings[]` even on exit 0.** The normal pass emits warning-class dimensions (`tableNotes`, `paywallRisk`, `unverifiedSource`, `fetchTrailMissing`, `acknowledgedWarnings`) in `warnings[]` without flipping `ok` to `false`. Triage them now — either fix the underlying issue or plan an `acknowledgedWarnings` opt-out (see step 6) — because step 6's `--strict` sweep promotes every unacked warning into the failure set and the same warning text then carries no extra per-table/per-source diagnostic context.
5. On failure, fix dimensions in `retryOrder[]` first. Treat root-cause dimensions and global hints before object-by-object cosmetic edits.
   - **`retryOrder[]` absent / empty?** It is omitted from the envelope whenever `failedDimensions` is empty — i.e. when there is nothing to retry. If `ok: false` but `retryOrder` is absent, you are in `--strict` mode with only unacked warnings remaining (see step 6); fall back to the dimensions listed in `summary.unackedWarningDimensions[]`.
   - **Both `retryOrder[]` and `summary.unackedWarningDimensions[]` non-empty (only possible under `--strict`)?** `retryOrder[]` covers only the failure-class dimensions; the warning-class dimensions are not in it. Fix the `retryOrder[]` dimensions first (they are the root-cause dimensions), then in the same edit cycle fix or `acknowledgedWarnings`-opt-out every dimension in `summary.unackedWarningDimensions[]` (per step 6). Rerun the strict sweep once after all of those edits — do not rerun after fixing only `retryOrder[]` or the unacked warnings will simply re-emit and look like new failures.
   - Honor the `retryPolicy` in [`rules.md`](references/rules.md) → *Agent policy (binding)*: at most **3** retries (`maxChapterRetries: 3`) on top of the initial run — i.e. up to 4 `check-chapter` invocations total per normal-pass cycle — and each retry must reduce the **failure count** (`requireMonotonicFailureDecrease: true`). "Failure count" here = the number of items currently making `ok: false`: in normal mode that is `issueCount` (size of `issues[]`, post-suppression); under `--strict` it additionally includes `summary.unackedWarningDimensions.length`. The agent enforces this — no script blocks a non-monotonic retry, but exhausting the budget without a strict decrease is the same blocker condition as below.
   - **Monotonic-decrease violation triggers the blocker immediately** (do not consume the rest of the budget). If retry N's failure count is `>=` retry N−1's, your fix strategy is not converging; further retries on the same approach waste budget. Surface the blocker, summarize what you tried in retries 1…N, and stop.
   - **Failure-count formula**: compare `issueCount + summary.unackedWarningDimensions.length` between successive validator JSON envelopes within the same pass. **Important: `summary.unackedWarningDimensions` is a deduped Set of *dimension names*, not a count of warning instances.** Five `tableNotes` warnings on different tables collapse to a single `tableNotes` entry; fixing four of them does not lower the count. To make progress against the monotonic-decrease rule, you must fix **every** instance of a warning dimension (or `acknowledgedWarnings`-opt-out the dimension as a whole — see step 6) before that dimension drops out of the count. Save each envelope before editing so the next retry can be compared against it; if the count does not decrease, stop per the blocker rule above.
   - If the budget is exhausted with failures still present, **stop drafting this chapter and surface a user-facing blocker message** naming (a) the chapter file, (b) the surviving failure dimensions from the last `check-chapter` JSON, (c) the **phase that exhausted** (`mid-flight normal pass` / `mid-flight strict sweep` / `convergence rerun`), and (d) per-chapter retry-budget exhaustion. Do not proceed to step 6 for this chapter, do not author `report-meta.yaml`, and do not run `finalize-report` — the run is incomplete until the user resolves the blocker (e.g. by providing additional sources or relaxing scope).
6. Once the normal pass is clean, run a **strict pre-finalization sweep**:
   `node .agents/skills/startup-research/scripts/check-chapter.mjs <reportFolder> <chapter.file> --strict --format json`
   - `--strict` adds any unacknowledged warning dimensions (e.g. `unverifiedSource`, `tableNotes`) to the failure set so `ok` flips to `false`. The warnings stay in `warnings[]` (they are *not* moved into `issues[]`); `summary.unackedWarningDimensions[]` lists the dimensions now contributing to the failure count, which is `issueCount + summary.unackedWarningDimensions.length`.
   - When a warning is genuinely non-actionable (e.g. `tableNotes` for a pure factual snapshot whose dimension's `defaultFix` explicitly tells you to acknowledge it), opt out by adding a top-level `acknowledgedWarnings: [{ dimension, reason }]` entry to the chapter YAML — see the schema in [`contracts.md`](references/contracts.md). `reason` must be at least 30 characters; only the warning-class dimensions listed in [`rules.md`](references/rules.md) → *Validator dimensions* → *`acknowledgedWarnings` opt-out* are eligible. Never use this to silence real failures — `finalize-report` reruns this strict sweep automatically and will refuse to publish if any chapter still fails it.
   - **Escalation when a warning is neither fixable nor ack-eligible.** Some warning dimensions are not in the `acknowledgedWarnings` opt-out list (e.g. `fetchTrailMissing`, `paywallRisk`), and a small number of cases are also structurally unfixable (e.g. an `unverifiedSource` warning where the cited URL is paywalled and `fetch-url` cannot retrieve it, but the source is the only adverse witness available). When you hit such a case before the retry budget is exhausted, **stop and surface a user-facing blocker mid-cycle** rather than burning the rest of the budget on retries that cannot converge. The blocker must name (a) the chapter file, (b) the offending dimension and the specific id/url/field, (c) why neither fix nor ack applies, and (d) the user-actionable resolution options (e.g. supply an alternative open-access source, drop the source and accept the lower coverage, escalate to the maintainer to add the dimension to the ack list). Do not author `report-meta.yaml` or run `finalize-report` for this chapter until the user responds.
   - **Strict starts a new baseline.** Do not compare the first strict sweep against the preceding clean normal pass (normal may be `0` while strict newly exposes warning-class failures). If the first strict sweep fails, save that strict JSON envelope as the baseline; subsequent strict retries must reduce the failure count from that strict baseline.
   - **Strict retry budget is per-chapter, applied across consecutive strict retries of the same chapter.** The `maxChapterRetries: 3` budget applies to fixes after the first strict sweep fails on a given chapter (each chapter has its own 3-retry budget; retries on different chapters are independent). Once that chapter's strict retry budget is exhausted, surface the same user-facing blocker message defined in step 5 (with phase = `mid-flight strict sweep`) and stop. `requireMonotonicFailureDecrease: true` applies between successive strict retries of the same chapter.
7. **Convergence rerun (parallel mode only — skip in sequential mode).** After every parallel chapter has landed on disk, walk every chapter **in `--list` order** and for **each chapter** run step 4 (normal pass) followed by step 6 (strict sweep) before moving to the next chapter; iterate within the convergence retry budget below until both sweeps clear on every chapter. (Per-chapter `normal→strict` ordering matches the mid-flight loop and lets the orchestrator stop at the first non-converging chapter without leaving the rest in a half-validated state.) `finalize-report` will run the strict sweep itself but won't fix root causes for you, so do this pass before authoring `report-meta.yaml`.
   - **Why required**: `netNewSources` and `crossChapterRefLeak` inspect on-disk siblings, so a chapter can pass in isolation and fail once a peer lands (or vice versa). Cross-chapter duplicates (`duplicateAnalysisCrossChapter`) and metric/key-fact drift are caught later by `check-cross-chapter` inside `finalize-report`, not by this rerun — the convergence rerun re-evaluates the per-chapter dimensions against the final on-disk sibling set so finalize does not have to fail on them.
   - **Convergence rerun gets a fresh retry budget — applied per chapter.** `maxChapterRetries: 3` resets for the convergence cycle and is interpreted **per chapter** (each chapter that surfaces convergence-only failures has its own 3-retry budget), not as a single 3-retry budget shared across the whole convergence pass. A chapter that consumed its mid-flight budget on non-transient fixes still has 3 retries available against the convergence rerun. Mid-flight and convergence budgets do not stack.
   - **Cross-phase ceiling.** The three phases (mid-flight normal, mid-flight strict, convergence) each get their own per-chapter `maxChapterRetries: 3`. Convergence is the **final** phase. If a chapter fails the convergence retry budget, surface the step 5 blocker (with phase = `convergence rerun`) and stop — do not start a fresh phase or repeat the convergence cycle.
   - **Mid-flight handling of transient failures during step 4** (before convergence): treat `netNewSources` / `crossChapterRefLeak` failures as transient — they reflect siblings that haven't landed yet. Do not burn `maxChapterRetries` rewriting them mid-flight.
     - **Failure set is *only* transient dimensions**: treat the chapter as "drafted, pending convergence". Skip step 5 (no retry needed) and skip step 6 (the strict sweep would just re-emit the same transient noise) for this chapter; move on to other parallel work and re-validate via this convergence rerun.
     - **Failure set mixes transient and non-transient dimensions**: fix only the non-transient ones within the retry budget (transient ones do **not** count toward `maxChapterRetries`); once the non-transient failures clear, proceed to step 6. The convergence rerun re-evaluates everything against the final on-disk sibling set.

> **Parallel worker contract.** If you fan out chapter drafting, pass each worker the `reportFolder`, `runId`, its chapter `order`/`key`/`file`/`letter`, the full `runtimeContext` for that chapter (including `runtimeContext.policy.retryPolicy`, which the worker must enforce since no script blocks non-monotonic retries), and the same `STARTUP_FETCH_LOG_PATH=.research-cache/<runId>/_fetch-log.jsonl` (resolve to an absolute path if any worker chdir's away from the repository root — the relative form only resolves correctly from the repo root). Each worker may write only `reportFolder/<runtimeContext.chapter.file>` and scratch files under `.research-cache/<runId>/`; it must not edit sibling chapters, `report-meta.yaml`, assembled artifacts, or git state. Each worker must return a brief summary so the orchestrator can detect non-converging workers; the orchestrator cannot audit per-worker retry counts otherwise. Required fields: `chapter.file`, `normalStatus` (`passed` / `failed`), `normalRetries` (count), `strictStatus` (`skipped` (transient-only failure deferred to convergence) / `passed` / `failed`), `strictRetries` (count), and `survivingFailureDimensions[]` (empty when both passed). The orchestrator must wait for every worker, verify every configured chapter file exists, then run the convergence normal+strict sweep before authoring `report-meta.yaml`.

> **Note: the fetch-url trail is shared across the run.** Every chapter writes into the same `STARTUP_FETCH_LOG_PATH` file (one JSON line per call, append-only); `check-chapter` only requires that each cited URL appear in the trail at least once. Parallel `fetch-url` calls do not need coordination; they only need to share the env var so `unverifiedSource` stays clean. (The subagent env-propagation rule for isolated runtimes is in *Runtime bootstrap* step 5; same rule applies here.)

## Finalization

Only start after all configured chapter artifacts exist and pass `check-chapter --strict` — step 6 in sequential mode, or step 7's convergence rerun in parallel mode.

1. Author `report-meta.yaml` from the `ReportMetaSchema` summarized in [`references/contracts.md`](references/contracts.md). It owns the final judgment, cover facts, and company profile fields.
2. Validate report meta shape:
   `node .agents/skills/startup-research/scripts/check-report-meta.mjs <reportFolder> --format json`
   - This step validates shape and enums only. Cross-references such as `coverFacts[].claimRefs` are resolved later by `build-report` against the consolidated `evidence.yaml`; dangling refs surface there, not here.
   - **`displayCompleteness` is a permanent published warning, not a gate.** `check-report-meta` has no `--strict` mode and never promotes warnings to failures. `displayCompleteness` (missing `companyProfile.<field>`, `coverFacts` items, claimRefs) rides in `warnings[]` and degrades the public display surface forever — read `warnings[]` even on exit 0, populate the named fields (or accept the documented `null` path), and rerun. There is nothing for `acknowledgedWarnings` to silence here.
3. Run finalization directly:
   `node .agents/skills/startup-research/scripts/finalize-report.mjs <reportFolder>`
   - For refresh: add `--refresh`. `--refresh-reason` is optional on this side — if omitted, `finalize-report` reuses the value cached in `.research-cache/<runId>/refresh-context.yaml` by `create-report-run.mjs` (it prints `[finalize-report] using cached --refresh-reason…` so the chosen value is auditable). You may re-pass `--refresh-reason` here as audit redundancy, but the value **must match the cache exactly** — `finalize-report` exits 1 on mismatch (it does **not** rewrite the cached value). New refresh runs cannot be created without a reason; only legacy/manual cache files should ever have an empty reason, and those must be fixed in `.research-cache/<runId>/refresh-context.yaml` before finalization.
   - **`--rebuild` is no longer required after chapter edits.** `finalize-report` auto-rebuilds the ledger whenever a chapter file's mtime is newer than `evidence.yaml` (it logs `[finalize-report] auto-rebuild:…`). Pass `--rebuild` only when you want to force a fresh consolidation regardless of mtimes. Cost note: `coverFacts[].claimRefs` use chapter-letter ids (`C<L>###`) that stay stable across rebuilds; rebuild itself never invalidates them. They go dangling only when an actual chapter edit removed or renamed the underlying claim id — if `build-report` reports a dangling ref after a rebuild, look at the chapter edit, not at the rebuild.
   - **`finalize-report` Exit `4`** (pre-flight): the report folder does not exist, `report-meta.yaml` is missing inside it, **or** any configured chapter file is missing when the strict sweep starts (stderr line: `[finalize-report] missing chapter file(s) before strict sweep: <files>`). The first case usually means the wrong path was passed; the second means you ran finalize before authoring `report-meta.yaml` — author it under the report folder and rerun; the third means finalize ran before every chapter from `--list` was on disk — return to *Chapter generation* and complete the missing chapters before retrying finalize.

Finalization is a sequential pipeline; it stops at the first failing step and exits with that step's code. Triage by step name printed to stderr (`[finalize-report] -> <step>` then `[finalize-report] <step> failed`):

| Step | Script | Fix target on failure |
|---|---|---|
| `check-chapter:<key>:strict` | `check-chapter.mjs --strict` | the named chapter (rerun the same command directly to iterate). A `missing chapter file(s) before strict sweep` stderr line means `finalize-report` was invoked before all configured chapters were drafted — return to *Chapter generation* and author every chapter listed in `--list` before retrying finalize. |
| `check-report-meta` | `check-report-meta.mjs` | `report-meta.yaml` shape/enum issues |
| `prepare-refresh` (refresh only) | `link-refresh.mjs --prepare-current` | ensure the prior `current` report exists, is finalized, and its `summary-card.yaml` matches the new report on **either** `company.name` (normalized) **or** `company.website` (normalized registrable domain) — the match is OR, not AND |
| `build-evidence-ledger` | `build-evidence-ledger.mjs` | a chapter `localEvidence` block (the script names the chapter and offending id) |
| `check-cross-chapter` | `check-cross-chapter.mjs` | metric drift, key-fact overlap, or duplicate analysis across chapter YAMLs; after editing chapters, rerun `finalize-report` (it auto-rebuilds the ledger when chapter mtimes are newer and re-runs the strict sweep) |
| `build-report` | `build-report.mjs` | a chapter or `report-meta.yaml` field that the assembler could not project |
| `check-report` | `check-report.mjs` | report-level gates (domain diversity, adverse-source distribution, paywall ceiling) |
| `link-refresh` (refresh only) | `link-refresh.mjs` | only fires after publishability passes. All abort messages start with `[refresh]`; common causes: (a) `report folder not found` / `not finalized` / `not under reports/` — the path passed to `finalize-report` is wrong; (b) `multiple current finalized reports match the new report company/domain` followed by `ambiguous refresh target …` — by the time link-refresh fires the new report is already finalized, so neither `create-report-run --refresh-of` (rejected post-finalization) nor `--resume` (the cache-rewrite path is not actually wired) recovers it; the only runtime-safe fix is to **edit `.research-cache/<newRunId>/refresh-context.yaml` and set `refreshOfRunId: <intendedPriorRunId>`** (the file is scratch, not subject to the hardRules ban on revision-field edits), then rerun `finalize-report --refresh`; (c) `<priorRunId> is already superseded by <otherRunId>` — refresh the current report (`<otherRunId>`) instead; (d) the script also re-runs `build-report.mjs` against the *prior* run's chapter YAMLs to reassemble its `summary-card.yaml` / `full-report.yaml`, so it fails when the prior chapters no longer satisfy the current schema/gates (e.g. after a schema change to a dimension the prior run didn't satisfy). For (d), triage by running `check-chapter --strict` against the prior folder and fixing what it surfaces. |

`finalize-report` exits **1** in two distinct families: (a) early-exit causes the script raises itself before launching any subprocess, and (b) pass-through codes from any inner `check-*` / `build-*` step that returned 1. To triage, read the stderr `[finalize-report] -> <step>` / `[finalize-report] <step> failed` lines first — when no `<step> failed` line precedes the exit, the cause is in the early-exit family below; when one does, rerun that step directly with `--format json` per the *Tip* further down for the structured fix hints.

Early-exit (no `<step> failed` line printed):
- `--refresh` is set, `--refresh-reason` is also passed on the CLI, and the value does not match the one cached in `.research-cache/<runId>/refresh-context.yaml`. The script does **not** rewrite the cache from the CLI value — omit `--refresh-reason` on `finalize-report` to reuse the cache (recommended), or pass the same string verbatim for audit redundancy.

> **Tip:** When `finalize-report` aborts, the failing inner step's output is whatever format that step uses by default — `check-chapter:<key>:strict` is invoked with `--format compact`, every other check/build step runs with the default `--format text`. Both are human-readable but lack structured `issues[].fix`, `objectFailures[]`, `globalHints[]`, `retryOrder[]`, etc. Rerun the failing step directly with `--format json` to get those structured fields, then re-run `finalize-report` to drive the pipeline forward — it **always restarts at the top of the pipeline** (there is no resume-from-failing-step), so every earlier check/build re-executes too. The only smart skip is `build-evidence-ledger`: finalize compares chapter mtimes against `evidence.yaml` and only rebuilds when a chapter is newer (or `--rebuild` is passed). Step coverage:
>
> - `check-chapter:<key>:strict` → `node .agents/skills/startup-research/scripts/check-chapter.mjs <reportFolder> <chapter.file> --strict --format json`
> - `check-report-meta` → `node .agents/skills/startup-research/scripts/check-report-meta.mjs <reportFolder> --format json`
> - `build-evidence-ledger` → `node .agents/skills/startup-research/scripts/build-evidence-ledger.mjs <reportFolder> --format json`
> - `check-cross-chapter` → `node .agents/skills/startup-research/scripts/check-cross-chapter.mjs <reportFolder> --format json`
> - `build-report` → `node .agents/skills/startup-research/scripts/build-report.mjs <reportFolder> --format json`
> - `check-report` → `node .agents/skills/startup-research/scripts/check-report.mjs <reportFolder> --format json`
> - `prepare-refresh` / `link-refresh` (refresh only) → `link-refresh.mjs` does **not** support `--format`; triage from the `[refresh] ...` stderr lines (which name the offending file/field) and rerun `finalize-report.mjs --refresh` after the fix. Do NOT hand-edit `revision:` to work around it — `link-refresh` owns `revision.status`, `revision.refreshOfRunId`, `revision.supersededByRunId`, and `revision.refreshReason` on both runs. Refresh disambiguation happens at create time via `create-report-run.mjs --refresh-of <priorRunId>`.

When `finalize-report` exits 0, summarize the run for the user using every field listed in [`rules.md`](references/rules.md) → *Agent policy (binding)* → `finalResponseFields`. ### Refresh runs

`--refresh` rewires the revision graph in addition to producing a normal report:

1. `create-report-run.mjs --refresh` finds the unique finalized `current` report for the same company and writes `.research-cache/<runId>/refresh-context.yaml`. **Company match is OR-semantics**: a prior report counts as the same company when its existing `summary-card.yaml` matches the *new* invocation's `<companyName>` / `--website` arguments on either normalized name **or** normalized registrable domain — either alone is sufficient. If more than one current report matches, create-run stops before writing the new folder and lists candidates; rerun with `--refresh-of <priorRunId>` so the correct prior report is selected before refresh context is cached. (At finalize time, `link-refresh.mjs` re-checks the same OR rule between the prior `summary-card.yaml` and the new `report-meta.yaml` `company.*` fields you authored.)
2. The chapter loader projects that file as `runtimeContext.runCache.refreshContext`. Read it for prior-run summary, score, recommendation, and key metrics — but treat every value listed in [`rules.md`](references/rules.md) → *Agent policy (binding)* → `volatileFacts` as stale and re-fetch it.
   - **How to detect refresh mode at runtime**: the loader always emits `runtimeContext.runCache` when `--report-folder` is supplied; it is `{ refreshContext: null }` for a fresh run and `{ refreshContext: {…} }` when `create-report-run.mjs --refresh` cached a prior summary. Use `runtimeContext.runCache?.refreshContext != null` as the boolean signal — do not infer refresh from the run folder name or argv.
3. `finalize-report.mjs --refresh` runs `link-refresh.mjs` in two distinct phases:
   - **Pre-assembly (`prepare-refresh` step, `link-refresh.mjs --prepare-current`)** — only writes the *new* run's `revision.status: current`, `revision.refreshOfRunId`, `revision.supersededByRunId: null`, and `revision.refreshReason` onto its `report-meta.yaml`. Runs after `check-report-meta` passes and before `build-evidence-ledger`. The prior run is untouched at this point.
   - **Post-publishability (`link-refresh` step, default mode)** — only fires after `check-report` passes. Flips the prior run's `revision.status` to `superseded`, sets its `supersededByRunId` back-pointer to the new runId, and reassembles its `summary-card.yaml` / `full-report.yaml` so cross-references stay consistent.
4. **Revision authoring rule**: omit the `revision` block from `report-meta.yaml` entirely. The two `link-refresh` phases above own every revision field (`status`, `refreshOfRunId`, `supersededByRunId`, `refreshReason`) on both the new and prior runs. If create-run reported ambiguous prior matches, resolve that before creating the folder with `--refresh-of <priorRunId>`; do not try to disambiguate later by pre-authoring `revision.refreshOfRunId`.
5. The `refreshReason` is required on the initial `create-report-run.mjs --refresh --refresh-reason "<text>"` invocation (quote the value — multi-word reasons without quotes parse the second word as an unknown flag and exit 1) and is recorded on the new and prior `revision.refreshReason` fields. `create-report-run` writes it to `.research-cache/<runId>/refresh-context.yaml`, and `link-refresh.mjs` copies it onto `revision.refreshReason` of both `report-meta.yaml` files during the two refresh phases. Do not re-pass it on `finalize-report.mjs` unless you want audit redundancy; the value must then match the cache exactly (mismatch = exit 1, no rewrite). `--resume` uses the cached reason from the original create-run invocation.
